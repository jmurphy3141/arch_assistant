# v1.8 One-Pager
## Orchestrator Skill Hardening + JEP Lifecycle (Acceptance Criteria)

- Date: April 22, 2026
- Scope: v1.8 requirements validation
- Implementation status: lifecycle contract + approved-lock revision flow in progress and wired to API/orchestrator/UI

## Purpose

Make orchestrator expert-policy enforcement deterministic and make JEP lifecycle behavior explicit, especially around approval lock and intentional revision.

## Non-Negotiables

1. `generate_jep` remains best-effort for draft velocity.
2. Required fields are always evaluated and reported.
3. Only `approved` JEPs are lock-protected.
4. Revision of approved JEP requires intentional user action.
5. Orchestrator uses hard policy pushback when blocked.
6. JEP responses embed lifecycle state and source context references + short snippets.

## Given / When / Then

1. Best-effort draft on partial info
- Given a customer with incomplete notes/Q&A
- When JEP generation runs
- Then a draft is produced with `[TBD]` where needed and `missing_fields` lists required gaps.

2. Required-field contract
- Given any generated JEP
- When response is returned
- Then required fields (`duration`, `scope_in`, `scope_out`, `success_criteria`, `owners`, `milestones`) are assessed and reflected in `jep_state.missing_fields`.

3. Embedded lifecycle contract
- Given JEP generate/latest responses
- When API payload is inspected
- Then `jep_state` is present with `state`, `is_locked`, `missing_fields`, `required_next_step`, and `source_context`.

4. Approved lock behavior
- Given JEP state is `approved`
- When user attempts direct regenerate/update
- Then orchestrator/API return hard policy stop with reason codes and required next step.

5. Intentional revision path
- Given JEP state is `approved`
- When user performs explicit revision request action
- Then state transitions to `revision_requested` and generation is allowed.

6. In-progress revision allowance
- Given JEP is not `approved` (draft/partial/in-progress)
- When user requests update/regenerate
- Then update proceeds without formal revision-request gate.

7. Source traceability
- Given any JEP generation
- When response metadata is inspected
- Then `source_context.references` and short `source_context.snippets` are included.

8. Orchestrator policy pushback quality
- Given a blocked JEP action
- When orchestrator responds
- Then response includes deterministic policy structure: `reason_codes`, `missing_fields`, `required_next_step`, `retry_instructions`.

9. Chat specialist continuity
- Given chat triggers JEP specialist path
- When tool metadata is inspected
- Then `tool_calls[].result_data.trace` includes JEP lifecycle/lock contract metadata.

10. No-regression guard
- Given full orchestrator suite
- When v1.8 changes are validated
- Then diagram/POV/WAF/Terraform specialist routes remain behaviorally intact.
